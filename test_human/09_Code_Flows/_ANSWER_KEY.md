## What "correct" looks like, per check

### Scenario A — authoring
- **A-1 routing:** the turn is handled by the CC **converse** path binding the code-flow tools
  (`create_code_flow`, `add_code_step`, `dry_run_code_flow`, `get_code_flow`, `list_code_flows`,
  `run_code_flow`, `schedule_code_flow`). CC log shows those tool calls; it must NOT show
  `delegate_to_builder_agent` for this turn. (The code-process shortcut sits at the TOP of
  `classify_intent`, ahead of the capability router.)
- **A-2 by-name creds:** step code contains `aihub.query("AIRDB", ...)` (or
  `aihub.connection("AIRDB")`) and `aihub.secret("AUTODEMO_SFTP")`. Grep the shown code for
  `10.0.0.6`, `ai_user`, `testuser`, `testpass`, `password` — **zero hits**.
- **A-3 honest dry-run:** the reply states the dry-run executed real code; per-step results with
  real row counts / file paths. A dry-run is NOT a simulation on this platform.
- **A-4 outputs:** CSV exists; file visible in the SFTP server's `/outgoing` (server terminal logs
  the STOR/upload; or check the `_sftp_test_server` data dir).
- **Value spot-check (optional):** run against AIRDB and compare 2–3 stores to the CSV:
  ```sql
  SELECT lm.store_name, COUNT(*) AS headcount
  FROM TS.employee_data ed JOIN TS.location_master lm ON ed.store_id = lm.store_id
  GROUP BY lm.store_name ORDER BY lm.store_name;
  ```
- **A-5 persistence:** `list my code flows` names **store-headcount**; it is a Workflows-table row
  with `kind='code_flow'` (also visible on the Workflows page).
- **A-6 schedule:** schedule rides the existing **workflow** job type (TargetId = the flow's
  workflow id); read-back shows next_run.

### Scenario D — input lint (0037)
- `add_code_step` / step-code update with `aihub.input("storefilter")` while the declared input is
  `store_filter` → rejected with an input-name mismatch error naming both strings. Agent retries
  with the corrected name in ≤1 turn. (Lint pattern matches `aihub.input('...')` usage against the
  step's declared inputs.)

### Checkpoints inside a Code Flow (aihub.checkpoint)
- If the agent adds an `aihub.checkpoint(...)` gate before an irreversible step (e.g. the upload),
  it **auto-approves** in a Code Flow with an honest log line: `[aihub] checkpoint auto-approved
  (not a supervised Automation run — human gates apply once this is promoted to an Automation): …`.
  This is expected — a Code Flow has no live AutomationRuns row to pause/resume against, so the gate
  can't hold; it takes effect once the process is promoted to an Automation (which Mission Control
  supervises). It is NOT a failure and NOT a silent skip. (A prior build 403'd here and killed the
  step before the upload — that's fixed.)

### Scenario E — fail-edge honesty
- With the SFTP server down, the upload step result is a **failure** (connection refused); the chat
  reply names the failed step + real error; any wired fail-edge/alert step executes. **Any** "✅
  uploaded"/"completed" claim while the server is down = automatic FAIL.

### Scenario F — builder honesty (0034, commit `5c8cded`)
- Ground truth: open workflow **truth-test** — persisted nodes are exactly
  **Database / Set Variable / File** (NO SFTP-anything node; the visual builder has no such type).
- The reply's step list is derived from those persisted nodes (a read-back emitted by the builder as
  a `workflow_saved` SSE event and composed deterministically CC-side): it lists exactly those
  3 steps, then discloses the **SFTP/FTP file transfer is NOT in this workflow** and steers to a
  **Code Flow** — wording like "NOT a complete replacement for what you asked".
- Builder service log: `workflow_saved read-back: id=<n>, status=success, node_types=['Database',
  'Set Variable', 'File']` (line is prefixed with a 🧾 emoji).
- FAIL conditions (the historical bugs): reply lists an "SFTP Upload" step, says "✅ verified"
  about the transfer, or asks for raw SFTP credentials.

### Scenario G — gates
- Role-1 run/delete of a `kind='code_flow'` workflow → HTTP 403 from the generic workflow routes
  (`_deny_code_flow_for_non_developer` in `app.py`). Developer+ succeeds.

## Log locations
- CC service log — tool-call lines (`create_code_flow`, `dry_run_code_flow`, ...), routing.
- Builder service log — `workflow_saved read-back:` (Scenario F).
- Main app log — pip install lines for `_pkg_cache` (Scenario C), 403s (Scenario G).
- SFTP test server terminal — upload/STOR lines (Scenarios A/E).
