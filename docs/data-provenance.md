# Data Provenance — v1 Contract

Status: **active** (Phase A, WS4). Introduced as part of the enrichment
foundation work, alongside the geocoder (WS2) and enrichment re-enable
(WS3).

This doc is the **schema contract** for the `_provenance` sibling map
that decorates Command Center data blocks. Anyone emitting a block that
contains an enriched field should read this. Anyone consuming a block to
render it (drawer, badges, audit walkback) should read this too.

## TL;DR

We picked **option 1: sibling map**. Original fields stay flat; a
parallel `_provenance` map keyed by field path carries the source/
timestamp/confidence for each enriched field. Existing consumers that
don't know about `_provenance` ignore it — zero ripple to the classic
renderer.

The alternative considered (wrapping each value as
`{value, source, ...}`) was rejected for v1 because every consumer would
have to read `.value`. That ripple is too costly for v1.

## Where it lives

`command_center_service/provenance.py` exposes:

- `ProvenanceEntry` — pydantic v2 model: `source`, `source_url`,
  `source_detail`, `timestamp`, `confidence`, `notes`.
- `Provenance` — convenience wrapper around
  `dict[str, ProvenanceEntry]` keyed by field path. Methods: `set`,
  `stamp`, `get`, `merge`, `downgrade_confidence`, `to_dict`,
  `from_dict`.
- `attach_to_block(block, prov)` — write helper that idempotently
  merges into `block["_provenance"]`.

## Block shape

```jsonc
{
    "type": "map",
    "title": "NYC",
    "center": [40.71, -74.00],
    "markers": [{ "lat": 40.71, "lng": -74.00, "label": "NYC HQ" }],

    "_provenance": {
        "markers[0].lat": {
            "source": "geocoder",
            "source_url": null,
            "source_detail": "nominatim",
            "timestamp": "2026-05-10T15:23:00+00:00",
            "confidence": 0.97,
            "notes": null
        },
        "markers[0].lng": { /* … */ }
    }
}
```

Two non-obvious rules:

1. **Field-path convention.** Dot notation for nested fields,
   `[i]` for list items, exactly the way you'd reach into the structure
   in JS (`block.markers[0].lat`). Keep them stable; the UI keys badges
   off these strings.
2. **Don't try to stamp every field in the system in v1.** Scope is:
   map markers (`lat`, `lng`), chart data points, table cells flagged by
   `_check_fabrication`. Anything else stays untracked until the
   pipeline matures.

## Source vocabulary (closed set)

| value             | meaning                                                |
|-------------------|--------------------------------------------------------|
| `db`              | Internal/business database, fetched by a data agent.   |
| `web_search`      | Live web search result (Tavily etc.).                  |
| `geocoder`        | Geocoding lookup (Nominatim / Mapbox / …).             |
| `model_knowledge` | LLM parametric knowledge — inherently uncertain.       |
| `user_input`      | Came from the human's message verbatim.                |
| `system`          | Generated by the agent harness (timestamps, ids).      |
| `unknown`         | Catch-all when source can't be determined.             |

Adding a new source value means updating both `VALID_SOURCES` in
`provenance.py` **and** the badge color map in `ops-room.js`.

## Confidence semantics

`confidence ∈ [0, 1]` or `null`.

- `null` → "we didn't measure" (distinct from "we measured and got 0.0").
- `geocoder` → Nominatim's `importance` field, clamped.
- `model_knowledge` → 0.5 default; the LLM can self-report a higher value
  but it's still untrusted relative to `db`/`web_search`.
- After `_check_fabrication` flags a field, confidence is lowered (never
  raised) and a `notes` line records *why*.

## UI surfacing (ops-room drawer)

Per field shown in the drawer, render a small badge using the source ➜
color mapping below. Hover/click reveals a tooltip with source name +
URL (if any) + timestamp + confidence + notes.

| source            | badge   | notes                                  |
|-------------------|---------|----------------------------------------|
| `db`              | green   | internal DB                            |
| `web_search`      | blue    | live web search                        |
| `geocoder`        | cyan    | geocoded                               |
| `model_knowledge` | amber   | LLM parametric                         |
| (low-confidence)  | red     | fabrication-flagged                    |
| (no entry)        | gray    | untracked                              |

See `ops-room.js: renderProvenanceBadge` for the actual rendering.

## Versioning

This is **v1**. The contract is intentionally narrow so we can extend
later (e.g. multi-source corroboration, evidence URLs into the trace
store). Add a `schema_version` key to `_provenance` only when we break
back-compat — for now the absence of the key implies v1.
